## 1. 产品定位

### 1.1 一句话

**打开网页就能玩的中文短局文字冒险:用自然语言行动,在天黑前找回走丢的猫。**

### 1.2 借鉴与不借鉴 AI Dungeon

| 借鉴 | 不做 |
| --- | --- |
| 自由输入「我要做什么」 | 无限世界、无结局 |
| 流式叙事、沉浸阅读 | 无状态约束的纯聊天 |
| 轻量网页即开即玩 | 角色卡社区、长线存档平台 |

核心差异(也是给玩家的卖点):**每局有限回合、可见状态、明确通关条件。** 这是"游戏"而不是"套壳聊天"。

### 1.3 在作品集中的角色

| 项目 | 角色 |
| --- | --- |
| AI Photo Studio CN | 垂直 AI 应用(生图/任务队列/成本控制) |
| Evidence Graph | 可追溯研究产品 |
| mcp-guardian | Agent 工具门禁 |
| **一局(本项)** | 可玩的轻量 AI 交互产品;展示"模型写故事、规则改分数"的约束式 LLM 设计 |

作品集叙事文案:

> 轻量中文互动小说:AI Dungeon 式自由输入,但每局有限回合、可见状态与明确通关条件。首发模组《找回走丢的猫》。

### 1.4 明确不做(首期)

- 账号 / 支付 / 模组市场
- 语音、3D、多人对战
- 无限续写、跨周长线云存档
- 同时做多个模组(引擎预留扩展点;内容只深做「找猫」)

## 3. 模组故事:找回走丢的猫

### 3.1 背景(给玩家看的)

傍晚,小区里,你发现自家的猫**「年糕」**不见了。
天快黑了。邻居还在楼下晃,物业亭还亮着灯,草丛和车库里可能有线索。

你的目标很简单:**在天黑前找到年糕,并把它带回家。**

### 3.2 为什么选这个故事

- 目标一句话能懂,不需要懂「推理」「生存」术语
- 场景封闭(一个小区),事件好设计、状态好收敛
- 胜负清晰,适合 20–40 分钟一局
- 语气温暖,演示友好、分享友好

### 3.3 角色(可对话)

| 角色 | 作用 | 关键信息 |
| --- | --- | --- |
| 你 | 玩家 | — |
| 门卫老周 | 知道进出与监控大致情况 | 指向绿化带;有条件开放监控线索 |
| 陈阿姨(遛娃邻居) | 可能看见猫的走向 | 指向车库或快递柜 |
| 快递员小吴 | 刚送完快递 | 提到纸箱里有动静 |
| 孩子豆豆(P1 可选) | 喜欢猫 | 可能喂过或吓跑过年糕 |

### 3.4 地点(可前往)

| 地点 | 可能发生 |
| --- | --- |
| 单元楼下 | 开场、问路 |
| 绿化带 / 草丛 | 发现脚印、铃铛、毛发 |
| 地下车库入口 | 有声音;暗处需要手电 |
| 物业亭 | 问监控、借手电 |
| 快递柜旁 | 纸箱、气味线索 |
| 自家门口 | 带回猫后的结局点 |

首期不做大地图:文字切换地点即可(「你来到绿化带」)。

## 4. 玩法设计

### 4.1 核心循环

```text
阅读当前叙述
  → 输入框用自然语言写行动(「去草丛找找」「问问门卫」)
  → 系统解析意图 + 规则引擎更新状态 + 生成下一段叙述
  → 看右侧状态(天色、线索、亲近感、剩余行动)
  → 重复,直到通关或失败
```

### 4.2 意图分类(引擎理解的行动类型)

自由输入,系统归一到 6 类意图:

| 意图 | 示例输入 | 引擎效果 |
| --- | --- | --- |
| 移动 `move` | 「去绿化带」「回物业亭」 | 切换地点,**不消耗行动**(与 §4.5 的"7-8 步通关"估算一致,该估算只数交互动作) |
| 搜证 `search` | 「仔细找找」「听听声音」 | 按地点+旗帜掉落线索,消耗 1 行动 |
| 对话 `talk` | 「问问那个穿红衣服的阿姨」 | 触发角色对话事件,消耗 1 行动 |
| 使用 `use` | 「用手电照角落」「打开纸箱」「撒猫粮」 | 检查前置旗帜,消耗 1 行动 |
| 呼叫 `call` | 「喊年糕——」 | 按亲近感给反馈,消耗 1 行动 |
| 结束 `finish` | 「带年糕回家」 | 需 `已找到`,触发结局 |

无法归类的输入 → 温和引导(「你想了想,还是先……」),**不消耗行动次数**(避免玩家因表达被惩罚)。

模糊指代要能落位:「问问那个穿红衣服的阿姨」→ `talk(陈阿姨)`。P0 由关键词/同义词表实现,P1 可换 LLM 意图解析。

### 4.3 玩家可见状态(只展示 4 项)

| 状态 | 含义 | 范围 |
| --- | --- | --- |
| **天色** | 时间压力 | 傍晚 → 黄昏 → 擦黑 → 天黑(每 3 次行动推进一档) |
| **线索** | 已掌握的信息条数 | 0–6 |
| **亲近感** | 离找到猫有多近 | 远 / 有动静 / 很近 / 已找到 |
| **剩余行动** | 本局行动配额 | 初始 12;用尽且未找到则失败 |

内部隐藏旗帜(不展示):`heard_meow`、`got_flashlight`、`knows_garage`、`cat_in_box` 等。

### 4.4 胜负条件

**通关(好结局)**

- 亲近感 =「已找到」且玩家执行 `finish`
- 结局页:重逢叙述 + 本局回顾(用时 / 行动数 / 关键线索链)

**失败(温和收尾)**

- 天色到「天黑」仍未找到;或行动次数用尽
- 结局文案方向:「天黑了,你在家门口放了猫粮和年糕的毯子。明天一早再找。」+「再来一局」

**坏结局(P1 可选,慎用)**

- 首期不做。

### 4.5 一局节奏

- 目标时长 **20–40 分钟**
- 最短通关路径约 **6 次**行动(问门卫→绿化带搜→问阿姨→物业借手电→车库照→开箱→回家为 7 步,含 finish);典型玩家 8–12 次
- 12 次行动配额 = 允许 4–6 次走弯路,有压力但不苛刻
- 失败可即刻「再来一局」重置

### 4.6 给玩家的一句话说明(界面文案)

> 你可以随便说想做什么,但时间在走、线索会累计。
> 找到猫并带回家,这一局才算通关——不是无限聊下去。

## 5. AI 职责边界(核心设计原则)

| AI 负责 | 规则引擎负责(不交给模型自由发挥) |
| --- | --- |
| 生动的中文叙述 | 天色推进、行动次数扣除 |
| 角色口吻与反应 | 胜负判定 |
| 把玩家句子理解成意图(P1) | 关键旗帜校验(如是否已拿手电) |
| 在已有线索范围内「像人一样回答」 | 线索表与结局触发 |

原则:**模型写故事,规则改分数。**

P0 阶段"AI"整体由 mock 实现:意图靠关键词表,叙述靠模板池(同事件多套文案随机)。玩法先跑通,再决定哪些环节值得换真模型。

## 6. 内容骨架(事件池)

用「事件卡片」组织内容,每张卡 = 触发条件 + 状态效果 + 叙述模板(≥2 套避免重复感)。

**开场**

- 天色=傍晚;亲近感=远;线索=0;行动=12

**早期(线索 0–1)**

- 问门卫老周 → 「往绿化带跑了」(线索+1)
- 喊名字 → 远处似有回应(亲近感可提一档,概率性)

**中期(线索 2–3)**

- 绿化带搜证 → 铃铛/毛发(线索+1,亲近感=有动静)
- 问陈阿姨 → 指向车库或快递柜(线索+1)
- 物业亭 → 借到手电(`got_flashlight`)

**后期(线索 4+)**

- 车库暗处:无手电 → 「太黑了什么都看不清」;有手电 → 听到猫叫(亲近感=很近)
- 快递柜旁纸箱 → 发现年糕(亲近感=已找到,`cat_in_box`)

**结局**

- `finish` → 通关叙述 + 回顾

防刷规则:同一线索不重复掉落;重复问同一人 → 短句「我刚才都说了…」且**仍消耗行动**(时间压力的一部分)。

## 7. 界面结构(产品层,非视觉定稿)

单页应用:

```text
┌─────────────────────────────────────────────┐
│  一局 · 找回走丢的猫          [再来一局]     │
├──────────────────────────┬──────────────────┤
│  叙事区(滚动段落)         │  天色   黄昏     │
│                          │  线索   2        │
│                          │  亲近感 有动静   │
│                          │  剩余行动 7      │
│                          │  当前地点 绿化带 │
├──────────────────────────┴──────────────────┤
│  [输入你想做的事…]              [行动]       │
└─────────────────────────────────────────────┘
```

- 移动端(≤768px):状态栏折叠为顶部一行胶囊
- 通关/失败进入结局页 → 「再来一局」
- 视觉方向后续按 frontend-design 流程单独定,不在本文档定稿

## 8. 分期计划

### P0 — 最小可玩版(先看效果)

1. 仅模组《找猫》
2. 自由输入 + 关键词意图解析 + 模板叙述(纯 mock,零 API 成本)
3. 可见四状态 + 通关/失败 + 结局回顾
4. 本地打开即玩,无需注册

### P1 — 接真模型与打磨

- 可选真实 LLM(叙述润色 + 意图解析;每局调用次数上限,超限回落 mock)
- 结局回顾展示「关键线索链」
- 模组列表壳(「找猫」可玩,其余灰显占位)
- 部署到公网(静态托管 + 一个 API 路由)

### P2 — 内容扩展

- 从 §8.4 候选池选第二个模组,只加配置与事件卡,不改引擎(验证引擎/内容分离设计)

### 8.4 候选模组池(仅在「找猫」验证通过后启动,一次只做一个)

| 模组 | 一句话卖点 | 备注 |
| --- | --- | --- |
| **找回走丢的猫** | 目标极清楚,老人小孩都懂 | **首发,当前唯一在做** |
| 相亲局翻车 | 中文味重,对话天然好玩 | 对话密集,适合验证角色口吻 |
| 电梯故障 60 分钟 | 场景封闭、状态少,两周够做完 | 单地点,引擎压力最小 |
| 春运抢票夜 | 文化差异强,作品集一句话就懂 | 时间压力玩法可复用「天色」机制 |
| 照顾植物一周 | 最轻,适合验证「自由输入 + 状态 + 结局」 | 多日制,需要引擎支持回合分段 |

明确约束:候选池只是路线图,**不并行开发**;每个新模组立项前先确认上一个的成功标准(§9)达成。

---

## 9. 成功标准

P0 完成后自测,以下全部成立才进入 P1:

1. 不看说明也能明白「要找猫、要赶在天黑前」
2. 用自然语言玩,不需要背指令(含模糊指代)
3. 能稳定打出一次通关、一次失败
4. 体感是「在玩游戏」,不是「在跟 ChatGPT 闲聊」
5. 一局不超过 40 分钟
